================================================================================
KEY FEATURES IMPLEMENTED
================================================================================

EVENT MANAGEMENT
  • Create events with name, seats, and future date
  • View all events with filtering and sorting options
  • Calculate available seats dynamically
  • Track active registrations per event

USER REGISTRATION
  • Register users for events
  • Prevent duplicate registrations (same user, same event)
  • Prevent overbooking with double-check validation
  • Cancel registrations and free up seats

VALIDATION & SECURITY
  • Event name uniqueness validation
  • Future date validation for events
  • Positive seat count validation
  • User name and event ID validation
  • Comprehensive error messages

DATA PERSISTENCE
  • JSON file-based database at data/db.json
  • Automatic data persistence between restarts
  • Consistent data format with two collections:
    - events: All event records
    - registrations: All user registrations

RACE CONDITION PREVENTION
  • Triple-check mechanism in registration endpoint
  • Prevents concurrent request seat count conflicts
  • Validates seat availability before and after registration
  • Safe for production use with single-threaded Node.js

API ENDPOINTS
================================================================================

EVENTS
  POST /events                        Create new event
  GET /events                         List all events
  GET /events?sort=date               Sort by date
  GET /events?upcoming=true           Filter upcoming
  GET /events/:eventId                Get specific event

REGISTRATIONS
  POST /registrations                 Register user
  GET /registrations                  List registrations
  GET /registrations?eventId=...      Filter by event
  GET /registrations?userName=...     Filter by user
  GET /registrations/:registrationId  Get specific registration
  PATCH /registrations/:id/cancel     Cancel registration

UTILITY
  GET /health                         Server health check

================================================================================
VALIDATION RULES
================================================================================

Event Creation
  ✓ Event name: Required, unique, non-empty string
  ✓ Total seats: Required, number > 0
  ✓ Event date: Required, valid ISO 8601 format, future date

User Registration
  ✓ User name: Required, non-empty string
  ✓ Event ID: Required, must exist
  ✓ Seat availability: Event must have available seats
  ✓ Duplicate check: User cannot register twice for same event

================================================================================
ERROR HANDLING
================================================================================

HTTP Status Codes Used
  201     Created (event, registration)
  200     Success (GET, PATCH)
  400     Bad request (validation error)
  404     Not found (event, registration)
  409     Conflict (event full, duplicate, already cancelled)
  500     Server error (unexpected)

Error Response Format
  {
    "success": false,
    "error": "Error title",
    "details": ["Detailed error message"],
    "message": "Additional context"
  }

================================================================================
SEAT COUNTING LOGIC
================================================================================

Available Seats = Total Seats - Active Registrations

Key Points:
  • Only "active" registrations are counted
  • "Cancelled" registrations are excluded
  • Seats freed immediately when registration is cancelled
  • Shown in event responses as "availableSeats"
  • Double-checked during registration to prevent overbooking

================================================================================
RACE CONDITION HANDLING
================================================================================

Problem: Multiple concurrent requests might exceed event capacity

Solution: Triple-Check Validation
  1. Check available seats (first validation)
  2. Check no duplicate registration (second validation)
  3. Final verification before database write (third check)
     - Re-query active registrations
     - Verify capacity not exceeded
     - Only register if seats available

Result: Guaranteed no overbooking, even with concurrent requests
